# 08. UI仕様（Phase 1-4）

## 目的

Phase 1の画面構成と操作感を固定する。本ドキュメントはdocs作成のみを目的とし、TypeScript実装、React実装、CSS作成はまだ行わない。

## 1. UIの基本思想

- 人間が最速で比較判断するための画面
- 検索結果を1件ずつ開かせない
- 上位4件を同時に見せる
- 重複・類題も消さず、見比べられるようにする
- iPad横向きを第一想定にする

## 2. 画面全体構成

- 上部：検索エリア
- 中央：上位4件カードエリア
- 下部：5件目以降の結果リスト
- 検索エリアは画面上部に固定する
- ソフトウェアキーボード表示時にも検索窓が隠れないことを重視する

## 3. 検索窓

- 画面上部に大きく配置
- 検索ボタンは置かない
- 2文字以上で即時検索
- 1文字以下では結果も予測も非表示
- プレースホルダー例：「用語・問題文・解説を検索」
- 入力中の視線移動を減らす

## 4. グレー予測

- 検索窓内、または検索窓と一体に見える位置に薄い文字で表示する
- 2文字以上で候補を出す
- 1文字以下では候補を出さない
- Enterは検索実行ではなく予測確定
- iPadではタップ確定も用意する
- 日本語IME変換中のEnterとは衝突しないよう実装時に注意する
- Phase 1では高度な候補生成ではなく簡易候補でよい

## 5. 上位4件カード

- 検索結果の上位4件を常時表示する
- iPad横向きでは2列×2行を基本とする
- 各カードに最低限表示する情報：
  - 分野
  - 難易度
  - 問題文
  - A/B/C/D
  - 正解
  - 解説
- 正解選択肢は視覚的に分かるようにする
- カードをクリックしなくても主要情報が読めることを重視する
- 文字量が多い場合はまず情報密度を調整し、安易に1件表示へ逃げない

## 6. 5件目以降リスト

- 上位4件の下に表示する
- 1行またはコンパクトカード形式でよい
- 表示項目：
  - 分野
  - 難易度
  - 問題文の先頭
  - 正解
- タップで詳細表示するかどうかはPhase 1 UI確認後に判断する
- MVPでは上位4件の体験を優先する

## 7. 状態別表示

- 空入力：何も出さない、または簡単な案内のみ
- 1文字入力：結果なし、予測なし
- 2文字以上で該当なし：「該当なし」を表示
- 検索中：1000問規模では原則ローディング表示不要
- CSV読み込み失敗：エラーを表示する

## 8. iPad横向き想定

- 横向きで検索窓、上位4件、リストが自然に見えることを優先する
- ソフトウェアキーボードで下部が隠れても、検索窓と上位カードの主要部分が見えることを重視する
- Magic Keyboard / Smart Keyboard利用時もEnter挙動を確認する
- 縦向きはPhase 1では最低限の折り返しでよい

## 9. Phase 1で作らないUI

- 学習履歴画面
- 正答率管理画面
- 設定画面
- CSVインポート画面
- 重複管理画面
- 詳細な問題編集画面
- Mac/Windows専用レイアウト

## 10. UI確認ポイント

- 4カードが本当に読めるか
- 文字サイズが小さすぎないか
- 検索窓とグレー予測がズレないか
- 日本語入力が邪魔されないか
- Enter確定が誤爆しないか
- 上位4件を見るだけで比較判断できるか
