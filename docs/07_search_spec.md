# 07. 検索仕様（Phase 1-3）

## 目的

Phase 1の検索挙動を固定する。本ドキュメントはdocs作成のみを目的とし、TypeScript実装はまだ行わない。

## 1. 基本方針

- 検索ボタンは作らない
- 入力中に即時検索する
- 検索時にDBは使わない
- 起動時に読み込んだGQuestion配列から、メモリ上N-gram Indexを作る
- Phase 1ではSQLite/FTS/Fuse.js/MiniSearchを使わない

## 2. 検索開始条件

- 有効な検索トークンが2文字以上になったら検索する
- 1文字以下では検索結果を出さない
- 1文字以下ではグレー予測も出さない
- 空白のみの場合も検索しない

## 3. 入力正規化

- 前後空白をtrimする
- 全角スペースを半角スペースに変換する
- 連続スペースを1つにまとめる
- 英字は小文字化する
- 日本語はそのまま扱う
- Phase 1ではひらがな/カタカナ変換、同義語展開、表記ゆれ補正はしない

## 4. 複数キーワード

- 半角/全角スペースで分割する
- 2文字未満のトークンは検索条件から除外する
- 有効トークンが0個なら結果なし
- 複数トークンはAND優先
- 全トークンに一致する問題を上位にする
- 一部トークンのみ一致する問題は下位候補として残してよい

## 5. N-gram方針

- Phase 1は2-gramを基本とする
- 必要に応じて3-gramは後で検討する
- 日本語の分かち書きは行わない
- 検索対象テキストから2文字ずつ切り出してindexを作る
- 1文字検索は仕様上行わない

## 6. 検索対象フィールド

**対象**：

- category
- difficulty
- question
- choices.A
- choices.B
- choices.C
- choices.D
- answer
- explanation

**対象外**：

- id
- sourceFile
- usedStatus

ただしid/sourceFile/usedStatusは表示やデバッグには使える。

## 7. フィールド重み付け初期案

| フィールド | 重み |
|---|---|
| question | 5 |
| category | 4 |
| choices | 3 |
| explanation | 2 |
| difficulty | 1 |
| answer | 1 |

Phase 1ではまずこの初期値で始め、UI確認後に調整する。

## 8. スコアリング初期案

- トークンがそのまま含まれる完全部分一致を高くする
- N-gram一致数を加点する
- 複数トークンすべてに一致した場合はボーナス加点する
- questionに一致した場合は強めに加点する
- 同点の場合はid順または読み込み順で安定化する

## 9. 検索結果の返却

- 検索結果はスコア順に並べる
- 上位4件をTopCardsに表示する
- 5件目以降をResultListに表示する
- 件数上限はPhase 1では仮に50件まででよい
- ただし内部的には全件スコアリングしてよい

## 10. 重複・類題の扱い

- Phase 1では重複判定を行わない
- 同じような問題でも削除しない
- 上位4件に似た問題が複数出てもそのまま表示する
- 人間が比較して判断する思想を優先する

## 11. グレー予測との関係

- グレー予測は検索結果そのものとは別機能とする
- Phase 1ではまず簡易候補でよい
- 検索仕様docsでは、予測候補の詳細ロジックは別docsまたは後続セクションで扱う
- Enterは検索実行ではなく予測確定に使う予定
- IME変換中のEnterとの衝突はUI実装時の注意点とする

## 12. 非対象

Phase 1では以下を行わない。

- 同義語展開
- search_tags利用
- term利用
- typo補正
- ひらがな/カタカナ正規化
- ローマ字検索
- SQLite FTS
- AIによる意味検索
- embedding検索
